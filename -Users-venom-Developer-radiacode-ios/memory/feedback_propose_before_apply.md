**Единственный триггер применения — сообщение пользователя, состоящее РОВНО из одного слова «применить».** Больше ничто не даёт права применять: ни «сделай», «давай», «ок», «применяй», «применить это», ни ответ на мой вопрос, ни кнопка/AskUserQuestion. Если сообщение ≠ ровно «применить» — я только показываю код/diff в чате и ничего не меняю. (Пользователь явно ужесточил правило этими словами: «чтобы ты применял код — я должен написать применить одним словом и больше никак».)

**Scope — любое мутирующее действие, не только Edit/Write:** git-операции (`init`, `add`, `commit`, `push`, ветки), `rm`/`mv`, и любые команды, меняющие состояние репо/ФС/удалёнки. Read-only (status, log, diff, ls, сборки, чтение файлов) — без гейта.

**PRE-FLIGHT перед каждым мутирующим действием:**
1. Я показал код/diff в чате?
2. Последнее сообщение пользователя — ровно одно слово «применить»?
3. Это именно то изменение, что было показано, без «довесков»?

Только при всех трёх — действую. Иначе стоп: показываю и жду «применить».

**Не является согласием:** ответ на мой предыдущий вопрос (например «я всё удалил», «no preference»), «да», «ок», «давай», кнопки. Got burned: запустил `git init` + commit на radiacode-kmp по такому ответу — пользователь возразил («я тебе не давал добро»).

**Каждое применение — отдельное «применить».** Одобрение не каскадируется: после одного «применить» следующее изменение требует нового «применить».

**Срочность не снимает правило.** Даже очевидный bugfix / «не собирается» — сперва показать код, потом ждать «применить».

**Исключения:**
- Memory writes (сохранение памяти) — без гейта, если не кодируют спорную политику (тогда спросить).
- `/pr-XXXX` flow — собственная единая точка подтверждения коммита/пуша.

**История:** сначала пробовал различать по смыслу глагола, потом button-based (AskUserQuestion). Пользователь в итоге свёл всё к одному явному текстовому триггеру — ровно слово «применить».
