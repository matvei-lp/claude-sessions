## Migration method (agreed with Matvei)
- Single-source: shared logic → `commonMain` (Kotlin); both platforms consume; iOS refactored onto it.
- **Minimal PRs, bottom-up** (leaf pure logic → … → features), **PROD-FIRST**: each PR = shared code + production app wired to it + Swift duplicate deleted. No dead shared code, no throwaway test screens.
- Keep iOS diffs tiny via Swift `typealias Old = Shared.New`. Always green/shippable both platforms.
- Naming decided **per-piece when porting**, not all upfront.
- 1 PR = 1 branch `Feature/RMP-XXXX` = 1 commit, base `develop`, reviewer `vxrossa`.

## Persistence = separate LATE track (do NOT port persisted types as leaves)
- DB = SwiftData, 10 schema versions (SwiftData → KMP SQLDelight/Room is its own heavy track).
- **Only DB-coupled device type = `Alarm.Level`** (Codable inside `EventDataObject.AlarmData`) → defer.
- Device identity/model are stored as **strings** (`deviceId`/`deviceModel`) and reconstructed at runtime — NOT persisted as structs → safe to port. Watch model-name **string parity** (Zero `displayName` "RadiaCode-Zero" vs advertised "RadiaCode Zero").
- Calibration stored as `[Float]` (not the struct); signal/config/firmware/settings are transient → all safe.

## Device-core target (kept deliberately simple)
- `DeviceConnection` (platform, holds the handle): `write(packets)`, `notifications: Flow<ByteArray>`, `state`, `disconnect()`. **BYTES ONLY — must know nothing about commands / RawStorage / isLastPackage.** (iOS `Peripheral` today is generic over `BluetoothCommand` and owns RawStorage + accumulation — a layering smell; when adapted to `DeviceConnection` it gets STRIPPED to a dumb byte pipe and that logic moves up to `DeviceClient`.)
- `DeviceClient` (commonMain) = the command-send manager + queue: `suspend send<R>(command): R`, serialized via `Mutex` (one command in-flight). Loop: frame → write → accumulate notifications into the command's RawStorage until `isLastPackage` → decode. ALL command/codec knowledge lives here, NOT in the transport.
- Commands know their own bytes (codec, pure Kotlin).
- Transport split is per-platform behind `DeviceConnection`: iOS keeps its proven Swift `Peripheral` (CoreBluetooth) as an adapter (don't rewrite/replace with Kable hastily — hardware-parity risk); Android = Kable. `DeviceTransmition` (iOS) → replaced by shared `DeviceClient`.
- **Models (DONE, RMP-26, `core/device/model/`):** `RadiaCodeDevice(identifier, model)` identity (`==` by identifier) · `RadiaCodeModel` (sealed; `invoke(rawName)` / `serialCode`) · `RadiaCodeFamily` · `BluetoothSignal` (`invoke(rssi: Int?)`) · `RadiaCodeDiscovery(device, signal, peripheralId)` = scan find, `==` by device (NOT a data class), `peripheralId` = Kable `adv.identifier` for connect/reconnect. So identity and scan-find are TWO types (Discovery holds a Device), not collapsed. (`DiscoveredDevice`/`ConnectedDevice`/`DeviceSession` rejected; `RadiaCodeDiscovery` accepted.)
- **Connected device** (later, at connect): a `RadiaCode`-ish entity enriched after connect, NO transport handle, enriched data in ONE optional bundle (e.g. `info: DeviceInfo?`). Final shape decided at connect time.
- `ConnectionController` + `DeviceSetup` (handshake) + Repositories (polling→Flows) + VMs — all just USERS of `send`; build later.

## Package layout (`core/device/`, agreed RMP-23)
`model/` data · `scan/` `DeviceScanner`/`KableDeviceScanner` + `RadiaCodeDiscoveryParser` · `connection/` `DeviceConnection`+connect (later). ## Codec ground-truth (from iOS `Services/BluetoothCommands/*`; all little-endian)
Request: `[type?4][op4][if writeVirtualString → len4, or single 0x00 when no payload][payload?]`, then `[LE u32 = len(data)] ++ data`, chunked by 18 bytes (ONE length word, then chunk). Response framing: Default storage (header excl. leading length word = 8 or 12; `isValid == 1`) vs System (header 4, no valid key). Per-command decoders; CP1251 strings. SetExchange hostID `0xFF12FF01`.
